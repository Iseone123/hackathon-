# ML System Design: Система генерации и приоритизации научных гипотез

## 1. Проблема

### 1.1. Контекст
На старте НИОКР-проектов по переработке хвостов обогащения специалисты формулируют гипотезы вручную, опираясь на личный опыт. Накопленные учебники, регламенты, схемы флотации и данные о составах хвостов не используются системно. Нет единого формата обоснования и сравнения гипотез — старт проектов затягивается, а нетривиальные комбинации реагентных схем и режимов остаются нераскрытыми.

### 1.2. Задача в ML-терминах
По запросу вида *«повысить извлечение золота из хвостов КГМК на 10% при доступном оборудовании: флотомашины ФМ-6.3»* и корпусу знаний (учебники, схемы флотации, регламенты, таблицы состава хвостов) сгенерировать **ранжированный список проверяемых гипотез**, каждая с:
- конкретным предложением (реагент / концентрация / режим / изменение схемы);
- механистическим обоснованием и ссылкой на источник (grounding);
- оценкой реализуемости, риска и ожидаемого эффекта на KPI;
- опционально — дорожной картой проверки (последовательность опытов, критерии успеха/провала).

### 1.3. Реальные входные данные

Система разрабатывается на конкретном наборе данных по флотационному обогащению:

| Тип данных | Описание | Формат |
|---|---|---|
| Составы хвостов | Элементный состав пульпы для 4 объектов (КГМК, НОФ Вкр, НОФ мед, ТОФ) | Excel (.xlsx) |
| Экспертные гипотезы | Гипотезы, сформулированные специалистами для тех же 4 объектов | DOCX |
| Схемы флотации | 12 схем технологических процессов | PNG + PDF |
| Регламенты / оборудование | Типовые перечни оборудования, регламенты процессов | PNG (скриншоты) |
| Учебники | 5 учебников по флотационному обогащению, металлургии золота/серебра, извлечению из упорных руд | PDF |

> **Ключевой актив:** 4 пары (Excel хвостов → DOCX с экспертными гипотезами) — это одновременно few-shot примеры для промптинга и размеченный ground truth для оценки. Система должна воспроизводить или превосходить эти гипотезы.

### 1.4. Онтология домена (минимально необходимая)

| Тип сущности | Примеры |
|---|---|
| **Хвосты** | состав (Au г/т, Ag г/т, Cu %, Fe %, S %), тип руды, дата пробы |
| **Реагент** | ксантогенат, аэрофлот, известь, сернистый натрий; концентрация г/т |
| **Режим / операция** | pH среды, температура, время флотации, схема (прямая / обратная / коллективная) |
| **Свойство / KPI** | извлечение Au %, концентрат Cu %, хвостовые потери |
| **Оборудование** | тип флотомашины, мельница, классификатор |
| **Эффект** | направление (повышает / снижает), величина, условия проявления |

Связи: `реагент —[при условии]→ режим —[даёт эффект]→ KPI на хвостах [тип]`.

### 1.5. Ключевые вызовы
| Вызов | Почему сложно |
|---|---|
| Разнородные форматы | Excel с числами, PDF учебников, PNG схем — три совершенно разных пайплайна обработки |
| Изображения схем | Схемы флотации — ключевой контекст, но обычный RAG их не читает |
| Галлюцинации LLM | Гипотеза без реального источника хуже, чем отсутствие гипотезы |
| Малый корпус | 5 учебников + 4 примера — недостаточно для дообучения, нужен умный промптинг |
| Оценка новизны | Нужно отличать пересказ известного от нетривиальной комбинации |

### 1.6. Пользователи и сценарий
Технолог / руководитель направления → загружает Excel с составом хвостов + формулирует цель и ограничения → получает 5–15 гипотез с обоснованием → сравнивает с экспертными → фильтрует и отправляет в работу. Итерация — **минуты**.

## 2. Архитектура (от простого к рабочему)

Принцип: каждый уровень строится поверх предыдущего. На хакатоне цель — уровни 0–2 как рабочее демо, уровни 3–4 — roadmap в презентации.

### Уровень 0. Baseline: few-shot + LLM без RAG
**Что делаем:**
- Берём 2 из 4 готовых пар (Excel → гипотезы) как few-shot примеры прямо в промпт.
- Новый Excel конвертируем в читаемый текст (pandas → markdown-таблица).
- Промпт: `[системный контекст домена] + [2 few-shot примера] + [новые хвосты] + "Сгенерируй N гипотез в формате..."`.

**Зачем:** собирается за 2–3 часа; даёт первые результаты; референс для сравнения; проверяет, что задача решается промптингом в принципе.

**Риски:**
- Нет ссылок на источники — все «факты» из параметрического знания модели.
- Ограничение контекстного окна при большом количестве few-shot примеров.
- Нет воспроизводимости и контроля hallucination.

---

### Уровень 1. RAG на учебниках + few-shot
**Что делаем:**
- Парсинг 5 PDF учебников (PyMuPDF / pdfplumber — не GROBID, это не статьи).
- Чанкинг с перекрытием → эмбеддинги (bge-m3 — лучший выбор для русского научного текста) → векторная БД (Qdrant / ChromaDB).
- Гибридный retrieval: BM25 + dense + reranker (bge-reranker-v2-m3).
- Промпт теперь содержит: few-shot примеры + **релевантные пассажи из учебников** + данные хвостов.
- Каждая гипотеза обязана цитировать конкретный пассаж (`[Учебник X, стр. Y]`).

**Плюсы:** появляется grounding — гипотезы опираются на реальные источники; снижаются галлюцинации; latency < 30 сек.

**Риски:**
- Наивный чанкинг рвёт таблицы реагентных режимов — нужен отдельный парсинг таблиц.
- BM25 плохо работает по числам (`«pH 8–9»`, `«150 г/т»`) — гибрид обязателен.
- Учебники описывают общие принципы, не конкретные объекты → гипотезы могут быть недостаточно специфичны.

---

### Уровень 2. Мультимодальный контекст + структурированный вывод (целевой для демо)
**Что делаем:**

**2a. Схемы флотации через vision:**
- Отправляем PNG схем в Claude / GPT-4o Vision с промптом «опиши технологическую схему: стадии, реагенты, потоки».
- Полученные текстовые описания добавляем в векторную БД как отдельный тип документа с метаданными `{тип: схема, объект: ...}`.

**2b. Регламенты и оборудование:**
- OCR скриншотов (pytesseract / easyocr) → текст в ту же БД.
- Альтернатива: vision-модель читает напрямую.

**2c. Структурированный вывод гипотез:**
Каждая гипотеза генерируется строго по схеме (structured output / function calling):
```json
{
  "id": 1,
  "hypothesis": "Добавка сернистого натрия 200 г/т на стадии I флотации...",
  "mechanism": "Na₂S депрессирует пирит и активирует сульфиды Cu/Au...",
  "expected_effect": "+8–12% к извлечению Au",
  "source": ["Учебник по флотации, стр. 142", "Схема флотации 3"],
  "feasibility": "высокая — реагент доступен, совместим с ФМ-6.3",
  "risks": "передозировка ведёт к депрессии целевых минералов",
  "score": {"novelty": 3, "impact": 4, "feasibility": 5, "risk": 2}
}
```

**2d. Ранжирование:**
- LLM-критик оценивает каждую гипотезу по 4 критериям (1–5).
- Итоговый скор = взвешенная сумма (веса настраиваются через UI — «режим экспертной настройки»).

**Плюсы:** полный grounding, прозрачный вывод, мультимодальный контекст, структура готова для экспорта.

---

### Уровень 3. Граф знаний (roadmap, не хакатон)
Извлечённые тройки `(реагент) —[эффект при условии]→ (KPI на хвостах типа X)` складываются в граф (Neo4j). Генерация гипотез — поиск непрямых связей (ABC-схема Свонсона). Визуализация подграфа как обоснование гипотезы.

**Почему не на хакатоне:** качество графа критически зависит от IE-пайплайна; на малом корпусе (5 учебников) граф будет разрежённым; нормализация сущностей на русских текстах — отдельная трудоёмкая задача. Оценка: 3–5 дополнительных дней после хакатона.

---

### Уровень 4. Суррогатная модель + байесовская оптимизация (roadmap)
На структурированных данных состав-хвостов → KPI обучается суррогатная модель (гауссовские процессы при малых данных, XGBoost при достаточном объёме). Генерация гипотез как контрфактуальный анализ: «минимальное изменение реагентной схемы → максимальный прирост KPI». Кандидаты прогоняются через RAG для механистического обоснования.

---

### Уровень 5. Агентная система + фидбэк-луп (roadmap)
Мультиагентный пайплайн: **Генератор → Критик → Novelty-checker** (поиск в открытых патентных базах) **→ Ранжировщик → Планировщик экспериментов**. Эксперт помечает гипотезы (принята / опровергнута / подтвердилась) → few-shot банк пополняется, веса ранжировщика дообучаются.

## 3. Метрики

### 3.1. Главная метрика: сравнение с экспертными гипотезами
Для каждого из 4 объектов (КГМК, НОФ Вкр, НОФ мед, ТОФ) — слепое сравнение топ-5 гипотез системы с экспертными гипотезами по шкале 1–5:

| Критерий | Описание |
|---|---|
| **Релевантность** | Гипотеза адресует заявленный KPI и ограничения |
| **Проверяемость** | Можно спланировать конкретный опыт с конкретными параметрами |
| **Научная состоятельность** | Механизм физически/химически правдоподобен |
| **Новизна** | Не дословный пересказ экспертной гипотезы из примера |

Целевой порог для демо: **≥3 из 5 гипотез** в топ-5 оцениваются на ≥4/5 по всем критериям.

### 3.2. Anti-hallucination: Groundedness
- **Faithfulness** (LLM-as-judge или RAGAS): доля утверждений в гипотезе, подтверждённых цитируемым пассажем.
- **Citation hit-rate**: % ссылок, где указанный источник действительно содержит приписанный факт (ручная проверка 10–15 гипотез).
- Gate-метрика: гипотеза без единого источника не попадает в финальный список.

### 3.3. Метрики retrieval
Для валидации поиска: 15–20 вопросов по домену × ручная разметка релевантных пассажей.
- **Recall@5 / Precision@5** — доля релевантных источников в top-5.
- Сравнение: dense vs hybrid (BM25 + dense) vs hybrid + reranker.

### 3.4. Системные метрики
- **Латентность**: от загрузки Excel до списка гипотез — цель < 2 минут для уровней 0–2.
- **Стоимость** на один запрос (LLM API вызовы).

> **Что убрали и почему:** Cohen's kappa / Krippendorff's alpha требуют ≥2 независимых оценщика — нереалистично для хакатона. MAE/RMSE предсказательной модели — модели ещё нет. Эти метрики перенесены в roadmap.

## 4. Технический стек

| Компонент | Выбор | Обоснование |
|---|---|---|
| Парсинг PDF | PyMuPDF + pdfplumber | Учебники — не статьи, GROBID избыточен |
| Парсинг Excel | pandas | Стандарт |
| Vision для схем/скриншотов | Claude claude-sonnet-4-6 / GPT-4o Vision | Схемы флотации сложны для OCR, vision-модели справляются |
| Эмбеддинги | bge-m3 (BAAI) | Лучшее качество на русском научном тексте, мультиязычный |
| Векторная БД | ChromaDB (хакатон) / Qdrant (прод) | ChromaDB — zero-infra для демо |
| Reranker | bge-reranker-v2-m3 | Совместим с bge-m3, русский |
| LLM | Claude claude-sonnet-4-6 | Длинный контекст, structured output, русский язык |
| Structured output | Pydantic + instructor | Гарантирует схему гипотезы |
| UI | Streamlit | Быстро, достаточно для демо |
| Экспорт | python-docx + reportlab | PDF/DOCX отчёт |

---

## 5. Сценарий демо (конкретный)

1. Пользователь загружает `Хвосты КГМК.xlsx` через UI.
2. Вводит цель: «Повысить извлечение Au на 10%, доступное оборудование: ФМ-6.3, бюджет реагентов ≤ 500 тыс. руб./мес».
3. Система возвращает 8 гипотез со структурированным обоснованием и ссылками на учебники/схемы.
4. **Рядом** отображаются экспертные гипотезы из `Гипотезы КГМК.docx` — наглядное сравнение.
5. Пользователь регулирует веса критериев (ползунки новизна / реализуемость / эффект) → список перестраивается в реальном времени.
6. Экспорт: PDF-отчёт + JSON с гипотезами.

## 6. Приоритизация для хакатона

| Приоритет | Что делаем | Когда готово |
|---|---|---|
| **P0** | Excel-парсер + few-shot baseline (ур. 0) — рабочий вывод гипотез | Первые 3 часа |
| **P0** | RAG на учебниках + structured output (ур. 1–2) | День 1 |
| **P1** | Vision-описание схем флотации → в БД | День 1 вечер |
| **P1** | UI на Streamlit + экспорт PDF | День 2 утро |
| **P2** | Сравнение с экспертными гипотезами в UI | День 2 |
| **P3** | Ранжирование с настраиваемыми весами | День 2, если останется время |
| **Roadmap** | Граф знаний, суррогатная модель, агентная система | Слайды в презентации |

## 7. Главные риски (сводно)

| Риск | Митигация |
|---|---|
| Галлюцинации в гипотезах | Gate-метрика: нет ссылки → гипотеза не выходит в финал |
| Гипотезы = пересказ few-shot | Явный запрет в промпте + оценка семантической дистанции до примеров |
| Плохой парсинг PDF учебников | PyMuPDF + ручная валидация 2–3 глав; отдельный парсинг таблиц |
| Схемы флотации нечитаемы для OCR | Vision-модель + fallback: ручное описание 3–4 ключевых схем |
| Нет ground truth для оценки | Уже есть: 4 пары (Excel → DOCX с экспертными гипотезами) |
| Не успеть на хакатоне | P0 готово через 3 часа; P1 — рабочее демо; P2–P3 — улучшения |
