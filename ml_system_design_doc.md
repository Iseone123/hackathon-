# ML System Design: Система генерации и приоритизации научных гипотез

## 1. Проблема

### 1.1. Контекст
На старте НИОКР-проектов в НИИ и промышленных лабораториях гипотезы формируются вручную отдельными экспертами. Накопленные архивы (статьи, патенты, отчёты об экспериментах) не используются системно, нет единого формата обоснования и сравнения гипотез, старт проектов затягивается.

### 1.2. Задача в ML-терминах
Построить систему, которая по запросу вида *«повысить жаропрочность сплава X на 15% при ограничениях по сырью/бюджету»* и корпусу знаний (статьи, патенты, внутренние отчёты, таблицы экспериментов) генерирует **ранжированный список проверяемых гипотез**, каждая с:
- обоснованием и ссылками на источники (grounding);
- предполагаемым механизмом влияния;
- оценкой новизны, рисков и ожидаемого эффекта;
- (опционально) дорожной картой проверки.

### 1.3. Ключевые вызовы
| Вызов | Почему сложно |
|---|---|
| Разнородные данные | PDF со сложной вёрсткой, таблицы составов, формулы, три языка (ru/en/zh) |
| Галлюцинации LLM | Гипотеза без реального источника = хуже, чем отсутствие гипотезы |
| Оценка новизны | Нужно отличать «пересказ известного» от «нетривиальной комбинации» |
| Нет размеченного ground truth | «Хорошая гипотеза» — субъективная экспертная оценка |
| Мало структурированных данных | Внутренних экспериментов может быть десятки–сотни, не тысячи |

### 1.4. Пользователи и сценарий
Исследователь / руководитель направления → формулирует цель и ограничения → получает 10–30 гипотез → фильтрует, корректирует, отправляет в работу (Jira/YouTrack). Итерация должна занимать **минуты**.

## 2. Предлагаемые решения (от простого к сложному)

Принцип: каждый следующий уровень строится поверх предыдущего. На хакатоне разумно довести до уровня 2–3, а уровни 4–6 показать как roadmap.

### Уровень 0. Baseline: LLM + контекст «в лоб»
**Что делаем:** весь релевантный корпус (или его срез) кладём в контекст LLM с промптом «сгенерируй N гипотез с обоснованием».

**Зачем:** референсная точка для всех метрик; проверка, что задача вообще решается промптингом.

**Риски:**
- Высокий уровень галлюцинаций, «источники» выдумываются.
- Ограничение контекстного окна — корпус не влезает.
- Нет воспроизводимости и контроля новизны.
- Стоимость/латентность при больших контекстах.

**Когда достаточно:** никогда как прод, но обязателен как baseline для сравнения.

---

### Уровень 1. Классический RAG
**Что делаем:**
- Парсинг PDF (GROBID для статей, unstructured/pymupdf для отчётов).
- Чанкинг → эмбеддинги (мультиязычная модель, напр. multilingual-e5 / bge-m3) → векторная БД (Qdrant/FAISS).
- Retrieval top-k → генерация гипотез LLM с обязательным цитированием чанков.

**Плюсы:** быстро собирается (1–2 дня), сразу даёт grounding и ссылки на источники — закрывает требование «прозрачность».

**Риски:**
- Наивный чанкинг рвёт таблицы, формулы, связки «состав → режим → свойство».
- Векторный поиск плохо работает по точным сущностям («0.3% Nb», «ГОСТ 5632») — нужен гибрид с BM25.
- Retrieval вытягивает похожее, а не *пробелы в знаниях* → гипотезы будут пересказом известного (низкая новизна).
- Качество на русскоязычных научных текстах у эмбеддеров ниже, чем на английских.

---

### Уровень 2. Структурированный RAG: извлечение сущностей + гибридный поиск
**Что делаем:**
- IE-пайплайн: извлечение сущностей и связей (материал, добавка, концентрация, процесс/режим, свойство, величина эффекта) — через SciSpacy + LLM-экстракцию по схеме (structured output / function calling).
- Метаданные (источник, год, тип документа, условия эксперимента) → фильтры при поиске.
- Гибридный retrieval: BM25 + dense + reranker (напр. bge-reranker).
- Генерация гипотез по шаблону: *{вмешательство} → {механизм} → {эффект на целевое свойство}*, каждая часть обязана ссылаться на извлечённые факты.
- Ранжирование гипотез LLM-критиком по критериям: новизна / реализуемость / эффект / риск, с настраиваемыми весами (режим «экспертной настройки»).

**Плюсы:** гипотезы становятся конкретными и проверяемыми; появляется интерпретируемая структура «факт → гипотеза».

**Риски:**
- Качество извлечения сущностей на смеси языков и в таблицах — главный источник ошибок (garbage in → garbage out).
- Нужна минимальная онтология домена (типы сущностей/связей) — требует времени эксперта.
- LLM-критик как ранжировщик смещён и нестабилен → нужна калибровка на экспертных оценках.

---

### Уровень 3. Граф знаний + Literature-Based Discovery (GraphRAG)
**Что делаем:**
- Извлечённые тройки (материал —[добавка]→ свойство, процесс —[влияет]→ свойство) складываем в граф (Neo4j).
- Генерация гипотез как **поиск непрямых связей** (ABC-схема Свонсона: A связано с B, B связано с C, связь A–C не описана → кандидат в гипотезы).
- Link prediction (эвристики соседства / node2vec / простые GNN) для оценки правдоподобия недостающих рёбер.
- LLM превращает найденный путь в текстовую гипотезу с механизмом и ссылками на рёбра-источники.
- Визуализация подграфа как обоснование (закрывает требование «графы, диаграммы влияния»).

**Плюсы:** это единственный уровень, который *систематически* находит новизну (пробелы), а не пересказывает корпус; сильная интерпретируемость — гипотеза = путь в графе.

**Риски:**
- Трудоёмкость: качество графа напрямую зависит от качества IE (уровень 2).
- Разрежённость: на малом корпусе граф даёт мало нетривиальных путей.
- Комбинаторный взрыв кандидатов A–C → нужен жёсткий скоринг/фильтрация.
- Нормализация сущностей (один сплав под пятью названиями) — отдельная боль.

---

### Уровень 4. Гибрид с предсказательными моделями
**Что делаем:**
- На структурированных данных экспериментов (состав + режим → свойство) обучаем суррогатную модель (градиентный бустинг; при малых данных — гауссовские процессы).
- Обогащаем открытыми датасетами (Materials Project, MatNavi) как prior.
- Генерация гипотез как оптимизация: байесовская оптимизация / контрфактуальный анализ («какое минимальное изменение состава даёт +15% к KPI при ограничениях»).
- Кандидаты от модели прогоняются через граф/RAG для механистического обоснования.

**Плюсы:** гипотезы получают количественную оценку ожидаемого эффекта с доверительным интервалом; прямая связь с KPI.

**Риски:**
- Мало внутренних данных → модель экстраполирует, а гипотезы как раз в зоне экстраполяции; нужна честная оценка неопределённости.
- Domain shift между открытыми датасетами и реальным производством.
- Легко получить «оптимум», нереализуемый технологически, — нужны жёсткие constraint'ы.

---

### Уровень 5. Агентная система + обучение на фидбэке
**Что делаем:**
- Мультиагентный пайплайн: Генератор → Критик (проверка механизма и grounding) → Novelty-checker (поиск по патентам/Semantic Scholar: не существует ли уже) → Ранжировщик → Планировщик экспериментов (дорожная карта: последовательность опытов, ресурсы, критерии успеха/провала).
- Фидбэк-луп: эксперт помечает гипотезы (принята / отклонена / подтвердилась в лаборатории) → данные идут в дообучение ранжировщика (learning-to-rank) и в few-shot примеры генератора.

**Плюсы:** закрывает «обучение на фидбэке» и дорожные карты из ТЗ; novelty-checker резко снижает долю «изобретённых велосипедов».

**Риски:**
- Латентность и стоимость (много LLM-вызовов на одну гипотезу).
- Сложность отладки и оценки многошаговой системы.
- Фидбэка мало и он медленный (лабораторная проверка — месяцы) → ранжировщик долго сходится.
- Риск переобучения на предпочтения конкретного эксперта.

## 3. Метрики

### 3.1. Метрики retrieval (уровни 1–3)
Нужна небольшая разметка: 30–50 запросов × релевантные документы (делается силами команды за день).
- **Recall@k / Precision@k** — доля релевантных источников в top-k.
- **MRR, nDCG@k** — качество ранжирования выдачи.
- Сравнение: dense vs BM25 vs hybrid vs hybrid+reranker.

### 3.2. Метрики достоверности генерации (anti-hallucination)
- **Faithfulness / Groundedness** (фреймворк RAGAS или LLM-as-judge): доля утверждений гипотезы, подтверждённых извлечёнными источниками.
- **Citation accuracy**: % ссылок, которые реально содержат приписанное им утверждение (проверка сэмплированием).
- **Hallucination rate**: доля гипотез с хотя бы одним неподтверждённым фактическим утверждением.

### 3.3. Метрики качества гипотез (экспертная оценка)
Основная метрика продукта. Слепая оценка экспертом по шкале 1–5 для каждой гипотезы:
- **Новизна** (не пересказ известного),
- **Проверяемость** (можно спланировать эксперимент),
- **Научная состоятельность** (механизм правдоподобен),
- **Релевантность** запросу и ограничениям.

Агрегаты: средний балл, **доля «пригодных» гипотез** (все критерии ≥4) на top-10, согласованность оценщиков (**Cohen's kappa / Krippendorff's alpha**), сравнение системы против baseline (уровень 0) на одних и тех же запросах.

### 3.4. Автоматическая оценка новизны
- **Семантическая дистанция** до ближайшего документа корпуса/патентной базы (слишком близко = не ново; слишком далеко = вероятно, бред → нужна «зона Златовласки»).
- **Ретроспективный тест (rediscovery)** — сильный аргумент для жюри: отрезаем корпус по времени (напр., до 2018), проверяем, «переоткрывает» ли система результаты, опубликованные позже. Метрика: % переоткрытых находок в top-k.

### 3.5. Метрики ранжирования гипотез
- **Spearman / Kendall tau** между ранжированием системы и ранжированием эксперта.
- **Precision@k**: доля гипотез в top-k, принятых экспертом в работу.

### 3.6. Метрики предсказательной модели (уровень 4)
- **MAE / RMSE / R²** на отложенных экспериментах (time-based split, не random!).
- **Калибровка неопределённости**: доля истинных значений внутри предсказанного 90%-интервала.

### 3.7. Системные и продуктовые метрики
- **Латентность**: время от запроса до первичного списка гипотез (цель: < 5 минут).
- **Стоимость** на один запрос (LLM-вызовы).
- Продуктовые (для питча, меряются в реальной эксплуатации): время до согласованного списка направлений (было недели → стало часы), % гипотез, дошедших до лаборатории, % подтвердившихся.

## 4. Рекомендация по объёму для хакатона

1. **Обязательно:** уровень 2 (структурированный RAG с гибридным поиском и цитированием) — рабочее демо.
2. **Сильно желательно:** мини-граф знаний (уровень 3) хотя бы на части корпуса + визуализация подграфа как обоснование гипотезы — это главный «вау-эффект» и закрытие требования интерпретируемости.
3. **В презентацию:** baseline-сравнение (ур. 0 vs ваше решение) по метрикам 3.2–3.3, ретроспективный тест новизны, roadmap на уровни 4–5.
4. **Экспорт:** PDF-отчёт + JSON/CSV задач — дёшево реализуется, закрывает требование интеграции.

## 5. Главные риски всего проекта (сводно)

| Риск | Митигация |
|---|---|
| Галлюцинации в гипотезах | Обязательный grounding, citation accuracy как gate-метрика |
| Гипотезы = пересказ известного | ABC-паттерны на графе, novelty-checker по патентам, зона семантической дистанции |
| Плохой парсинг PDF/таблиц | GROBID + отдельный пайплайн для таблиц, ручная валидация выборки |
| Нет ground truth | Экспертная разметка малой выборки + ретроспективный тест |
| Не успеть на хакатоне | Строгая приоритизация: ур. 2 → демо, ур. 3 частично, ур. 4–5 слайдами |
